# jvng. 가상 구매 플로우 QA 리포트

**작성일**: 2026-02-21  
**범위**: 상품 상세 → 장바구니 → 결제 → 주문 완료, 관리자 주문 노출, 배송 추적·FAQ·보안

---

## 1. 가상 구매 시나리오 실행 요약

### 1.1 플로우 검증

| 단계 | 경로 | 검증 내용 |
|------|------|-----------|
| 상품 상세 | `/shop` → 상품 클릭 → PDP | 품절 시 "SOLD OUT" 비활성, 장바구니 담기 시 수량 선택 가능 |
| 장바구니 | PDP "장바구니에 담기" → `/cart` | 수량 +/- 시 **합계 금액 실시간 반영** (state 기반 계산) |
| 결제 | "Proceed to Checkout" → `/checkout` | 서버 가격·재고·품절 검증 후 포트원 결제창 → 주문 저장 → 재고 차감 |
| 주문 완료 | 결제 성공 후 | 게스트: 주문 번호 + 이메일 안내 / 회원: 주문 번호 + 주문 내역 링크 |
| 관리자 | `/admin/orders` | `orders` 테이블 전체 조회, Realtime 구독으로 신규 주문 즉시 반영 |

### 1.2 시나리오 결과

- **상품 상세**: 품절 시 장바구니 담기 비활성, 재고 있으면 수량 선택 후 담기 가능.
- **장바구니**: `updateQuantity` 호출 시 `cart` state 갱신 → `totalPrice` 재계산으로 합계 실시간 반영됨.
- **결제**: `publicTable('products').select('id, price, name, stock_quantity, is_manual_soldout')` 로 서버 기준 검증 후 결제 진행. 재고 부족/품절 시 결제 단계에서 차단.
- **데이터**: 주문은 `orders` insert 후 `deduct_stock` RPC로 재고 차감. 재고 차감 실패 시 해당 주문 상태를 `취소됨`으로 업데이트하도록 수정함.

## 2. 문제점(이상한 점) 분석

### 2.1 로직 오류

| 항목 | 상태 | 비고 |
|------|------|------|
| 재고 없는데 구매 | ✅ 방지됨 | Checkout에서 `isSoldOut`·`insufficientStockProduct` 검사 후 결제 진행 |
| 수량 변경 시 합계 | ✅ 실시간 반영 | `CartPage`의 `totalPrice = cart.reduce(...)` 로 매 렌더마다 계산 |
| 재고 차감 실패 시 주문만 남는 문제 | ✅ 수정함 | 주문 insert 후 `deduct_stock` 실패 시 해당 주문 `status = '취소됨'` 으로 업데이트 (게스트/회원 공통) |

### 2.2 사용자 경험(UX)

| 항목 | 상태 | 비고 |
|------|------|------|
| PDP 수량 선택 | ✅ 수정함 | 상품 상세에 수량 선택기 추가 (1~재고 상한 또는 99), 재고 개수 표시 |
| 결제 버튼 위치 | ✅ 적절 | "장바구니로" / "결제하기" 나란히 하단 고정 |
| 에러 메시지 | ✅ 수정함 | "이름, 기본 주소, 전화번호" → "이름, 주소, 전화번호" 로 문구 통일 |
| 회원 주문 완료 후 | ✅ 수정함 | 회원도 주문 완료 화면에서 주문 번호 표시 + "주문 내역" 링크 제공 (기존에는 `/cart?order=success` 로만 이동) |

### 2.3 데이터 무결성

| 항목 | 상태 | 비고 |
|------|------|------|
| orders 저장 | ✅ 검증 | 결제 금액은 서버 `priceMap` 기준, 수량 상한 99 적용 |
| 재고 차감 실패 시 | ✅ 수정함 | 주문 insert 후 재고 차감 실패 시 해당 주문을 `취소됨`으로 업데이트해 "주문만 있고 재고는 그대로"인 상태 방지 |
| 관리자 주문 노출 | ✅ 수정함 | 회원 주문에도 `order_number` 저장, 관리자 목록/CSV에서 `order_number ?? id` 로 표시, 검색에 `order_number` 포함 |

## 3. 해결책 및 코드 수정 요약

### 3.1 PDP 수량 선택 (ProductDetailPage.jsx)

- **추가**: `addQty` state, 재고 상한 `maxQty` (1~99 또는 `stock_quantity`).
- **추가**: 수량 +/- 버튼 및 숫자 입력, 재고 개수 표시.
- **변경**: `handleAddToCart`에서 선택 수량만큼 `addToCart` 반복 호출, 토스트에 "N개 추가" 표시.

### 3.2 결제 페이지 에러 메시지 (CheckoutPage.jsx)

- **변경**: "이름, 기본 주소, 전화번호를 모두 입력해주세요." → "이름, 주소, 전화번호를 모두 입력해주세요."

### 3.3 회원 주문 번호 및 완료 화면 (CheckoutPage.jsx)

- **추가**: 회원 주문 시 `order_number = generateOrderNumber()` 부여 후 insert.
- **추가**: 회원 결제 성공 시 `orderSuccessData = { orderNumber, isGuest: false }` 로 주문 완료 화면 표시 (장바구니로 자동 이동 제거).
- **변경**: 주문 완료 블록을 게스트/회원 공통으로 두고, 회원은 "주문 내역" 링크, 게스트는 "주문 조회" 링크 제공.
- **변경**: `cart.length === 0` 일 때도 `orderSuccessData` 가 있으면 완료 화면 유지하도록 early return 순서 조정.

### 3.4 재고 차감 실패 시 주문 취소 (CheckoutPage.jsx)

- **게스트**: `orders.insert(...).select('id').single()` 로 삽입된 행 id 확보. 재고 차감 실패 시 `orders.update({ status: '취소됨' }).eq('id', guestOrderRow.id)`.
- **회원**: `orders.insert(...).select('id').single()` 로 id 확보. 재고 차감 실패 시 동일하게 `status: '취소됨'` 업데이트.
- **메시지**: "주문은 접수되었습니다" 대신 "재고가 부족하여 주문이 취소되었습니다" 등으로 통일.

### 3.5 관리자 주문 목록/검색/CSV (AdminOrdersPage.jsx)

- **표시**: 주문번호 컬럼에 `order.order_number || order.id` 표시.
- **검색**: `order_number` 포함 검색.
- **CSV**: "주문번호" 컬럼에 `o.order_number || o.id` 사용.

### 3.6 배송 추적 스테퍼 (OrderTrackingStepper.jsx)

- **추가**: `STATUS_TO_STEP`에 `취소됨: 0` 매핑 (취소 주문도 0단계로 표시).

## 4. 최종 점검: 배송 추적·FAQ와 구매 과정

### 4.1 배송 추적 그래프

- **위치**: `OrderTrackingStepper` / `OrderTrackingBlock` 은 **주문 내역(`/orders`)·주문 조회(`/order-lookup`) 페이지**에서만 사용.
- **결론**: 결제 플로우(`/cart` → `/checkout`)에는 배송 추적 UI가 없어 **구매 과정을 방해하지 않음**.

### 4.2 FAQ 모달

- **위치**: `FAQ` 컴포넌트는 **랜딩 페이지 섹션** 및 **`/faq` 전용 페이지**에서만 사용.
- **결론**: 결제·장바구니 페이지에 FAQ가 없어 **구매 과정을 방해하지 않음**.

## 5. 보안 이슈 확인 및 조치

| 항목 | 내용 | 조치 |
|------|------|------|
| 결제 금액 | 클라이언트 장바구니가 아닌 **서버 products 가격**으로 총액 계산 후 결제·주문 저장 | 이미 적용됨 |
| 재고/품절 | 결제 전 `products` 조회로 `is_manual_soldout`·`stock_quantity` 검증 | 이미 적용됨 |
| 장바구니 데이터 | `CartContext`에서 `sanitizeProduct`·`isValidProductId`로 화이트리스트 및 prototype 오염 방지 | 이미 적용됨 |
| 주문 저장 | `items`에 서버 `priceMap` 적용, 문자열 길이 제한(name 200, image 2048 등) | 이미 적용됨 |
| RLS | orders/products/profiles 등 Supabase RLS 정책으로 역할별 접근 제한 | 설정에 따름 |

**추가 권장**: 결제 완료 후 노출되는 주문 번호·이메일은 민감 정보이므로, 필요 시 주문 완료 페이지를 짧은 시간 후 자동 리다이렉트하거나, URL에 주문 id를 넣지 않는 방식 검토.

## 6. 요약

- **가상 구매 시나리오**: PDP → 장바구니 → 결제 → 주문 완료까지 로직·데이터 흐름 검증 완료.
- **로직**: 재고/품절 검증, 수량 변경 시 합계 실시간 반영, 재고 차감 실패 시 주문 취소 처리 반영.
- **UX**: PDP 수량 선택, 회원 주문 완료 화면·주문 번호 표시, 에러 메시지 문구 정리.
- **데이터 무결성**: 회원 `order_number` 저장, 관리자 목록/검색/CSV 반영, 재고 실패 시 `취소됨` 업데이트.
- **배송 추적·FAQ**: 구매 플로우와 분리되어 있어 방해 없음.
- **보안**: 금액·재고·입력값 검증 및 RLS 기준으로 현재 구조상 문제 없음.

위 수정 사항은 해당 컴포넌트/페이지에 이미 반영되어 있습니다.
